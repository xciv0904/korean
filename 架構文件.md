# 韓文情境口說練習網站 — 架構文件

**版本範圍:MVP(內容庫 + 跟讀練習)**
**目標使用場景:飯店櫃檯應對 / 日常約會口說**
**技術基底:延用 Caption Note(캡션노트)架構 — Vite + React + Vercel**

---

## 1. 專案目標

打造一個讓使用者可以依「情境」瀏覽韓文常用句、聽範例發音、錄下自己的口說並回放比對的練習網站。第一階段不做語音辨識自動評分,先確保「內容完整 + 跟讀流程順暢 + 可持續練習(SRS)」。

不做的事(明確排除於 MVP):
- 語音辨識自動評分(Phase 2)
- AI 情境對話模擬(Phase 3)
- 使用者帳號系統 / 雲端同步(先用本機儲存,未來可加)

---

## 2. 核心資料模型

### 2.1 情境分類(Category)結構

```
Domain(領域)
 └─ Category(情境分類)
     └─ ScenarioSet(情境組,例如「入住登記」)
         └─ Sentence(單句,含變化例句)
```

兩大 Domain:
- `hotel_frontdesk`(飯店櫃檯)
- `daily_dating`(日常約會)

### 2.2 Sentence 資料結構(核心 JSON schema)

```json
{
  "id": "hotel_checkin_001",
  "domain": "hotel_frontdesk",
  "category": "check_in",
  "scenarioSet": "확인 및 등록",
  "korean": "예약자 성함이 어떻게 되세요?",
  "chinese": "請問訂房人姓名是?",
  "formality": "formal_honorific",
  "romanization": "yeyakja seonghami eotteoke doeseyo?",
  "audioUrl": "/audio/hotel_checkin_001.mp3",
  "notes": "櫃檯標準敬語問句,可替換為「전화번호가/여권 번호가 어떻게 되세요?」",
  "alternates": [
    { "korean": "성함이 어떻게 되세요?", "usage": "更口語簡短版" }
  ],
  "tags": ["敬語", "詢問資訊"],
  "difficulty": 1
}
```

欄位設計重點:
- `formality`:標記敬語等級(`formal_honorific` / `polite` / `casual_banmal`),飯店櫃檯幾乎固定 formal,約會情境會混用,方便未來依等級篩選練習。
- `alternates`:同義替換,避免使用者只會死記單一句型。
- `audioUrl`:先用 TTS 批次生成(見第 5 節),之後可換真人錄音提升自然度。

### 2.3 使用者練習紀錄(本機儲存,IndexedDB)

```json
{
  "sentenceId": "hotel_checkin_001",
  "practiceCount": 3,
  "lastPracticedAt": "2026-07-18T10:00:00Z",
  "userRecordings": [
    { "timestamp": "...", "audioBlobKey": "idb_key_xxx" }
  ],
  "srs": {
    "interval": 3,
    "easeFactor": 2.3,
    "nextReviewAt": "2026-07-21T00:00:00Z"
  }
}
```

`srs` 欄位結構直接沿用 Caption Note 既有的間隔重複邏輯(同一套演算法,只是把「單字卡」換成「情境句卡」)。

## 3. 功能模組(MVP 範圍)

### 3.1 情境瀏覽(Browse)
- 依 Domain → Category → ScenarioSet 階層瀏覽
- 每張句卡顯示:韓文、中文、羅馬拼音(可切換顯示/隱藏)、敬語標籤
- 搜尋 / 篩選(依 formality、關鍵字、tag)

### 3.2 跟讀練習(Shadowing)— 核心功能
流程:
1. 播放範例音檔(TTS 或未來真人錄音)
2. 使用者按錄音鍵,用 `MediaRecorder API` 錄下自己念的版本
3. 錄完自動停止或手動停止,立即可播放「範例 vs 自己」雙軌比對
4. 使用者自評(滿意 / 需加強)→ 寫入 SRS 紀錄,決定下次複習時間
5. 錄音檔存在 IndexedDB(不上傳雲端,保護隱私也省成本)

介面建議:單句卡片式,左右滑動或上下鍵切換下一句,類似 Caption Note 的單字卡體驗,讓兩個 app 操作邏輯一致、使用者不用重新適應。

### 3.3 複習排程(SRS Dashboard)
- 首頁顯示「今日待複習句數」
- 複習模式:只跳出到期的句子,練習完畢自動排下一次複習時間
- 依 Domain 分開統計(飯店 vs 約會可能練習頻率不同,例如她若準備面試會集中衝刺飯店類)

### 3.4 進度總覽
- 各 Category 完成度(已練習句數 / 總句數)
- 連續練習天數(streak,沿用 Caption Note 既有的 streak check-in 機制)

## 5. 技術選型

| 項目 | 選擇 | 理由 |
|---|---|---|
| 框架 | Vite + React + TypeScript | 延用 Caption Note 既有技術棧,降低維護成本 |
| 錄音 | `MediaRecorder API`(瀏覽器原生) | 免額外套件,所有現代瀏覽器支援 |
| 範例發音 | 建置時批次生成 TTS(建議用 Naver Clova TTS 或 Google Cloud TTS 韓文語音,音質較 Web Speech API 內建自然) | 韓文 TTS 品質差異大,Web Speech API 內建語音生硬,不適合當練習範本 |
| 本機儲存 | IndexedDB(用 `idb` 套件簡化操作) | 錄音檔案較大,localStorage 容量不足 |
| 部署 | Vercel(比照 Caption Note) | 一致的部署流程 |
| SRS 演算法 | 沿用 Caption Note 現有邏輯 | 不用重新設計、驗證過的邏輯 |

**不建議在 MVP 階段導入語音辨識**,因為:
1. Web Speech API 的韓文辨識精準度不穩定,容易造成錯誤的挫折感(明明念對了卻被判錯)
2. 付費 API(如 Google STT)會拉高開發與維運成本
3. 跟讀+自我評估本身就是有效的口說練習方法,不做自動評分不影響核心學習效果

## 7. 開發階段建議

**Phase 1(本文件範圍)**
- 情境庫 + 瀏覽介面
- 跟讀練習(錄音/播放/比對)
- SRS 複習排程
- 進度總覽

**Phase 2(未來擴充)**
- 語音辨識回饋(評估用 Naver Clova Speech 或 Whisper API)
- 真人錄音取代 TTS 範例音檔
- 依難度/場景自訂練習組合

**Phase 3(未來擴充)**
- AI 情境對話模擬(用 Claude API,模擬「假客人」或「約會對象」即時對話)

## 8. 交給 Claude Code 時的建議提示重點

1. 先建立 `src/data/` 下的 JSON 檔案結構與範例資料(可先用 5-10 句佔位,驗證元件邏輯後再擴充全部句庫)
2. 優先完成 `ShadowingPractice.tsx` 核心流程並手動測試錄音/播放,這是風險最高的技術環節(瀏覽器錄音權限、音檔格式相容性)
3. SRS 邏輯直接參考/複製 Caption Note 專案中對應的 hook 與演算法檔案,不要重新設計
4. TTS 音檔生成建議寫成獨立 build script,而非執行期即時呼叫,避免每次載入頁面都要等 API
